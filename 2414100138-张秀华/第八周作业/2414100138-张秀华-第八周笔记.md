1.MVVM模式由三个关键组件组成：
①Model（模型）：表示应用程序的数据和业务逻辑，与UI完全无关。模型可以是简单的数据对象，也可以是复杂的业务领域模型。
②View（视图）：定义UI的结构、布局和外观，是用户与应用程序交互的界面。在MVVM中，视图是被动的，它通过数据绑定从ViewModel获取数据并显示。
③ViewModel（视图模型）：作为View和Model之间的中介，负责处理View的所有显示逻辑和用户交互逻辑。ViewModel暴露Model的数据和命令，使它们易于View进行绑定。


2.LiveData
2.1
LiveData遵循观察者模式，当底层数据发生变化时，LiveData会通知Observe对象，进而去Observe对象中更新UI操作，方便了开发者持续去监听数据的更新状态而更新UI
观察者会绑定到LifeCycle对象，并关联其生命周期，随多绑定的对象的生命周期的销毁而进行自我清理

2.2
ViewModel层：
private MutableLiveData<String> data = new MutableLiveData<>();
public LiveData<String> getData() {return data;}
public void setData(String newData) {data.setValue(newData);}

View层：
//初始化ViewModel
private UserViewModel viewModel;
viewModel = new ViewModelProvider(this).get(UserViewModel.class);
//观察
viewModel.userName.observe(this, { name ->
            。。。//数据发生变化这里会观察到，可进行处理
        })

ViewModel 持有 LiveData 对象，将数据的存储和更新逻辑封装在 ViewModel 中，而 UI 层只需要观察 LiveData 的变化，通过调用 getData() 方法获取 LiveData 对象并进行观察，处理完的数据用setValue方法将其设置值给LiveData，数据发生变化时，LiveData会通知Observe对象，进而去Observe对象中更新UI操作

2.3
为什么使用ViewModel:屏幕旋转、语言切换、字体大小改变 → Activity 会重建，重新执行 onCreate()，但 ViewModel 不重建，数据不丢、不重复请求网络。
UI 状态与数据的单向流动：Model -> ViewModel -> View



3.1  
View —— 视图层（XML + Activity/Fragment）
显示界面 + 接收用户点击：
负责
初始化和绑定控件（ViewBinding）
设置 RecyclerView、Adapter、LayoutManager等
接收用户交互（按钮点击、列表滑动、下拉刷新等）
观察 ViewModel 中的 LiveData 变化
根据 LiveData 结果更新 UI（刷新列表、显示文字、显示图片）
显示纯 UI 提示（Toast、Dialog、加载动画）
处理生命周期相关的 UI 操作（比如 onPause 暂停动画）

不可以：读写数据库、直接创建或持有 Retrofit、OkHttp 实例



3.2
ViewModel —— 视图模型层（调用Model层拿到数据，处理逻辑）
管理存放数据 + 处理业务逻辑：
负责：
持有并管理业务数据（MutableLiveData<T>）
调用 Repository 拿数据
处理数据业务逻辑（比如过滤、排序、分页、缓存处理）
把数据放进 LiveData 给 UI 观察(设置值： mNewsState.setValue(false);)
响应 View 层的用户操作（比如 refreshNews() 方法）

不可以：直接读写 SharedPreferences、数据库（这些交给 Model 层），处理UI


3.3
Model —— 数据层（数据源）
提供数据（网络、数据库、缓存）
负责提供数据来源：
发起网络请求（Retrofit 接口、OkHttp 封装）
本地数据库（Room、SQLite 操作）
本地缓存（SharedPreferences、文件读写）
定义数据实体类（比如 NewsBean、UserBean）
封装数据获取逻辑（比如统一处理网络请求、解析 JSON、异常处理）
提供给 ViewModel 统一的数据访问接口（比如 NewsRepository）
数据持久化（把网络数据存到本地数据库



Repository 属于 Model 层，是数据的「中转站、封装层」。
ViewModel 只指挥要数据，不亲自操作数据库和线程
UI 业务逻辑 → 放 ViewModel
跟数据源、存取数据、线程、数据来源有关 → Repository（Model 层）
